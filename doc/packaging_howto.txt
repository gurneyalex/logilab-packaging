Empaquetage d'un paquet conforme au "Standard Source Tree"
----------------------------------------------------------

Pour commencer, il est important de voir la documentation du fameux 
`standard source tree`_ avant de passer à la suite de ce document.


Création d'un tarball avec distutils
````````````````````````````````````
La première étape de la distribution d'un paquet Python est de rendre celui-ci
distribuable à l'aide de Distutils_ (cela n'est pas nécessaire pour les produits
Zope). Ce paquet fournit un fichier *setup.py* générique, tirant les
informations nécessaire du fichier *__pkginfo__.py*. Vous n'avez donc
théoriquement qu'à copier le setup.py générique et à écrire un fichier
*MANIFEST.in*. Pour en savoir plus ou si vous avez des besoins un peu exotique,
e vous laisse lire la documentation_ de distutils. Une fois cela effectué, il
est de bon ton de tester que ça marche...

1. **python setup.py sdist** à la racine du paquet.

2. vérifier que l'archive générée (dans le sous-répertoire *dist*) contient les
   fichiers désirés.  

3. extraire l'archive (tarball) dans un répertoire temporaire.

4. **python setup.py install** dans le répertoire extrait de l'archive
   (voir les options d'installation avec **python setup.py install --help**).

5. vérifier que les tests / éxécutables installés fonctionnent correctement.

L'utilitaire preparedistrib_ vous aide en faisant des vérifications et diverses
actions préparant la distribution. Il est donc fortement conseillé de l'éxécuter
avant d'effectuer une distribution. Une fois le tarball généré et vérifié,
utiliser l'utilitaire tagpackage_ pour mettre un tag dans le CVS correspondant
à la distribution que vous venez d'effectuer.

Création / mise à jour des fichiers de configuration debian
```````````````````````````````````````````````````````````
Pour créer et mettre à jour les fichiers de configuration nécessaire à la
construction d'un paquet debian, vous pouvez utiliser debianize_. Celui-ci va
créer tout les fichiers nécessaires (si votre package n'est pas trop exotique),
a part le fichier changelog qu'il faut créer et maintenir à la main. debianize_
prend principalement en considération les variables suivantes du fichier
`__pkginfo__.py`_ : 

- debian_name

- debian_maintainer

- debian_maintainer_email

- debian_handler

- architecture_dependant

- pyversion

- subpackage_of

Il ne vous reste ensuite plus qu'à taper **debianize** à la racine du paquet
pour qu'il crée ou mettes à jour les fichiers existants, en fonction de ces
variables et du contenu du paquet.


Test du paquet debian
`````````````````````
Pour tester rapidement votre paquet debian :

1. **debianize --orig prepare** à la racine du paquet, cela va vous créer le
   répertoire de construction ainsi que l'archive .orig.tar.gz contenant le
   package sans les fichiers debian.

2. **fakeroot make -f debian/rules binary** à la racine du répertoire de
   construction.

Cette méthode permet de tester que votre paquet se construit bien de manière
beaucoup plus rapide qu'avec pbuilder, qui va construire le paquet dans un
environnement construit spécifiquement pour le paquet (ce qui prend du temps...)
Si vous affectez la variable d'environnement ``DEBUILDER`` à **fakeroot make -f
debian/rules binary**, vous pouvez utilisez buildpackage_ pour automatiser ces
deux étapes (**buildpackage logilab/pyreverse deb** si on reprend l'exemple).

Une fois le paquet construit, il faut vérifier :

1. son contenu avec **dpkg --contents <monpackage.deb>**

2. son installation avec **dpkg -i <monpackage.deb>** 

3. que les tests / éxécutables installés fonctionnent correctement

4. sa déinstallation avec **dpkg -r <monpackage.deb>** 


Construction du paquet final
````````````````````````````

Une fois que tout marche, vous devez utiliser makedistrib_ pour construire le
tarball et le paquet debian. Il est important d'utiliser ensuite l'utilitaire
tagpackage_ pour mettre les tags dans le CVS.



Debianization d'un paquet externe
---------------------------------

Pour "debianizer" un paquet externe, a priori non conforme au standard source
tree, il faut spécifier toutes les informations nécessaires dans le fichier
*__pkginfo__.py*, y compris celles spécifiées comme étant optionelles dans le
document décrivant les variables du fichier `__pkginfo__.py`_. A partir de là,
tout devrait fonctionner de la même manière que pour nos paquets (évidemment
cela suppose dans le cas des packages python que le setup.py fonctionne !).
